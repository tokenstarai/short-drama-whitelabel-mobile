# Open Source Release Notes

## Default License

The mobile template is prepared for Apache-2.0 release. Add the repository-level license before public GitHub publishing if the whole monorepo is released; otherwise keep `mobile/` as the licensed package boundary.

## What Can Be Published

- Flutter source under `mobile/lib`.
- Public sample configs under `mobile/assets/config`.
- Template docs under `mobile/docs`.
- Android/iOS platform template files, excluding ignored local/generated files.
- App-level iOS privacy manifest `ios/Runner/PrivacyInfo.xcprivacy`, while it stays limited to template-default public declarations.
- Shared iOS entitlements file `ios/Runner/Runner.entitlements` for Sign in with Apple, while tenant team ids, provisioning profiles, and certificates stay outside Git.
- Example prototype screenshots generated from synthetic posters and demo data.
- Original generated launcher icons for the four template flavors.
- CI checks for static analysis, widget tests, prototype layout overflow tests, native config validation, Android debug APK builds, Android release APK/AAB builds, release manifest generation, store handoff manifest generation, unsigned iOS debug/release flavor builds, and secret scanning.
- Store-assets package generated by `scripts/export_store_assets.py`, containing per-flavor listing drafts, localized copy, review notes, data-safety starter facts, and publish-safe screenshots for tenant store-submission handoff.
- iOS CI handoff package generated by `scripts/export_ios_ci_handoff.py`, containing public workflow trigger, unsigned artifact names, Xcode scheme/xcconfig paths, tenant Apple action checklist, and no-credential boundary.
- iOS CI artifact evidence generated by `scripts/import_ios_ci_artifacts.py` after downloading unsigned GitHub Actions `Runner.app` artifacts.
- Store-signing handoff package generated by `scripts/export_store_signing_handoff.py`, containing placeholder iOS export-options and Android signing templates without real signing material.
- Store-publish config package generated by `scripts/export_store_publish_config.py`, containing tenant-fillable App Store, Google Play, and Android direct release configuration templates without provider credentials.
- Store-submission evidence template and guide generated by `scripts/import_store_submission_evidence.py`, containing only public TestFlight, Play internal, or direct-distribution status fields and no signing material.
- GitHub-ready template zip generated by `scripts/export_open_source_template.py`, after reviewing the package manifest and no-secret scan.

## What Must Not Be Published

- Tenant Edge HMAC secrets.
- OAuth client secrets.
- Stripe, PayPal, wallet, bank, or crypto private keys.
- Cloudflare API tokens, Stream signing keys, R2 credentials, or webhook signing secrets.
- Third-party app screenshots, copied marks, or trade dress assets.

Tenants must update `ios/Runner/PrivacyInfo.xcprivacy` before store submission if they add native analytics SDKs, advertising SDKs, tracking domains, new accessed API categories, or collection of data beyond the template defaults. They must also enable the Sign in with Apple capability on their own Apple Developer App ID to match `ios/Runner/Runner.entitlements`.

## Public Template Boundary

Templates are style presets, not brand clones:

- `hongguo_inspired`: light theater, red/orange charts, grid discovery.
- `douyin_inspired`: dark vertical feed, immersive player, compact unlock.
- `hippo_inspired`: channel theater, VIP emphasis, structured discovery.
- `reelshort_inspired`: overseas poster-forward drama, coins/subscription framing.

Tenants may override app name, colors, logo, supported locales, legal links, target region, and store compliance mode from the tenant portal export manifest.

## Android Release Artifacts

Run this from `mobile/` to produce a tenant-handoff Android App Bundle and a public artifact manifest:

```bash
./scripts/build_flavor.sh hongguo android release appbundle
./scripts/write_release_manifest.sh
./scripts/write_store_handoff_manifest.py
./scripts/scan_release_artifacts.py
./scripts/android_runtime_smoke.py all --launch-emulator
./scripts/ios_build_matrix.py all
./scripts/export_ios_ci_handoff.py
./scripts/import_ios_ci_artifacts.py
./scripts/export_store_assets.py
./scripts/export_store_signing_handoff.py
./scripts/export_store_publish_config.py
./scripts/import_store_submission_evidence.py
./scripts/export_completion_unblocker.py
./scripts/mobile_completion_closure.py
./scripts/export_open_source_template.py
./scripts/mobile_completion_audit.py
```

The release manifest is written to `build/release-manifests/mobile-artifacts.json` and contains only public release metadata plus SHA256 checksums. The store handoff manifest is written to `build/release-handoff/mobile-store-handoff.json` and combines public flavor config, build commands, artifact references, legal URLs, compliance-filtered payment visibility, OAuth/deep-link callback registration metadata, App Store/Google Play product registration metadata, native capability registration metadata, store review declarations, distribution channel readiness, store submission metadata, publish-safe screenshot asset references, and tenant-owned Apple/Google/direct-distribution checklist items. `paymentVisibility` shows configured providers, app-visible providers, providers hidden by the selected store compliance mode, and whether external payments are allowed. `authCallbackRegistration` lists each flavor's app URL scheme, `auth` host, enabled social OAuth callback URIs, expected callback query parameters, and native config files that own the scheme; provider credentials still stay in Tenant Edge/API Worker secrets. `storeProductRegistration` lists IAP or Play Billing product rows for store-compliant builds, the Tenant Edge receipt verification endpoint, and tenant actions for replacing template product ids while keeping App Store shared secrets and Google Play service-account credentials server-side. `nativeCapabilityRegistration` derives iOS and Android capability checklists from auth and payment providers, including Sign in with Apple, IAP, Play Billing, custom URL schemes, OAuth SHA fingerprints, direct-distribution signing, privacy manifest, entitlements, app icons, and native config paths. `storeReviewDeclarations` gives tenants a starter answer map for App Review, Play Console Data safety, and Android direct distribution declarations: digital-content payment policy, external payment visibility, account deletion, Sign in with Apple, user-generated content, privacy manifest, and direct-distribution payment duties. `distributionChannelReadiness` maps each flavor to its primary release channel, TestFlight build command and local Xcode blocker, Play internal AAB artifact, Android direct APK/AAB artifacts, and tenant-owned signing actions. `storeSubmission` gives tenants a public default listing starter, per-locale listing drafts for the configured supported locales, review notes, data-safety reminders, store-account actions, and eight prototype screenshot paths per flavor with SHA256, size, and PNG dimensions without replacing final App Store Connect or Play Console questionnaires. `android_runtime_smoke.py` can add local emulator/device proof that all four release APKs install and launch; it writes only ignored runtime evidence under `build/runtime-smoke/`. `ios_build_matrix.py` adds unsigned iOS debug/release build evidence for all four flavors on a full-Xcode machine, or writes an ignored `blocked` report with exact missing-toolchain details. `export_ios_ci_handoff.py` adds a public GitHub Actions handoff for remote unsigned iOS builds when local Xcode is unavailable; `import_ios_ci_artifacts.py` imports downloaded unsigned `Runner.app` artifacts into auditable public metadata. These CI artifacts still do not replace the full-Xcode completion gate. `export_store_signing_handoff.py` adds no-secret placeholder signing templates for tenant Apple/Google/direct distribution setup. `export_store_publish_config.py` adds no-secret tenant-fillable release configuration templates for App Store Connect, Play Console, and Android direct handoff. `import_store_submission_evidence.py` adds a blocked-by-default public evidence template and guide for TestFlight, Play internal, or direct-distribution status after tenant signing is done; tenants must not paste certificates, provisioning profiles, upload keystores, service-account JSON, OAuth credentials, payment keys, webhook secrets, bank credentials, crypto keys, or Cloudflare tokens into it. `mobile_completion_closure.py` reruns the public import/audit chain after new CI or tenant store evidence lands and writes the current `canClaimComplete` decision plus remaining blockers; it can resolve the GitHub repository from `--repo`, `GH_REPO`, `GITHUB_REPOSITORY`, or git `origin`. Build outputs stay under ignored `build/` directories. Tenants must replace the template debug signing configuration with their own store signing setup outside Git before submission.

`mobile_completion_audit.py` writes `build/completion-audits/mobile-app-completion.json` so release reviewers can distinguish completed template evidence from local iOS environment blockers. It also writes `build/release-handoff/mobile-tenant-release-package.json`, a public tenant handoff package that links the store handoff manifest, Android artifact manifest, store assets package, iOS CI handoff, store signing handoff, store publish config templates, store submission evidence/template, open-source docs, per-flavor release channels, OAuth callback metadata, store product metadata, tenant-required publishing actions, and no-secret boundaries.

## iOS CI Handoff Export

Run this from `mobile/` when local full Xcode is unavailable but a tenant or maintainer can use GitHub Actions macOS runners:

```bash
./scripts/export_ios_ci_handoff.py
```

The script writes `build/ios-ci-handoff/mobile-ios-ci-handoff.zip` and `build/ios-ci-handoff/ios-ci-handoff-manifest.json`. It records the manual `workflow_dispatch` trigger, `ios-build` matrix, unsigned artifact names, native metadata files, the local artifact download/import command, and tenant Apple account actions for all four flavors. It contains no signing material, Apple team ids, provisioning profiles, OAuth credentials, payment credentials, or Cloudflare credentials.

After the workflow completes, download and import the four `mobile-<flavor>-ios-unsigned` artifacts with:

```bash
./scripts/download_ios_ci_artifacts.py --repo <owner/repo>
```

The command selects the latest successful `mobile-flutter.yml` run by default. Add `--run-id <run-id>` when importing a specific run. It writes the artifacts under `build/ci-ios/`, then creates `build/ios-ci-evidence/ios-ci-artifacts.json` and per-flavor Info.plist snapshots. It validates bundle ids and display names for the downloaded unsigned `Runner.app` artifacts and records only public metadata. Use `--dry-run` to inspect the planned `gh run download` commands without contacting GitHub.

## Completion Closure Refresh

Run this after importing new iOS CI artifacts or tenant store-submission evidence:

```bash
./scripts/mobile_completion_closure.py
```

Use `--run-id <run-id>` to pin a specific GitHub Actions run, `--store-submission-source <path>` for a tenant-filled public evidence file, and `--strict` when CI should fail unless `canClaimComplete` is true. If `--repo` is omitted, the script uses `GH_REPO`, `GITHUB_REPOSITORY`, or git `origin`; without a repository it refreshes local artifact evidence only. The script writes `build/completion-closure/mobile-completion-closure.json` and `.md`; it contains command status, audit summary, and blocker next actions only.

## GitHub Template Export

Run this from `mobile/` before publishing the mobile template as a standalone GitHub repository or release artifact:

```bash
./scripts/export_open_source_template.py
```

The script writes `build/open-source/short-drama-whitelabel-mobile.zip` and `build/open-source/open-source-template-manifest.json`. It uses a publish-safe allowlist and excludes local build outputs, generated Flutter files, CocoaPods caches, `android/local.properties`, signing material, `.env` files, and secret-like values. The manifest records every packaged path, size, SHA256, required entries, package SHA256, and no-secret scan result so maintainers can review the exact GitHub upload boundary before publishing.

Run this after the open-source package is ready when a maintainer needs the exact GitHub publication commands without embedding credentials:

```bash
./scripts/export_github_publish_handoff.py
```

The script writes `build/github-publish/github-publish-manifest.json`, `github-publish-guide.md`, and `github-release-notes.md`. It records the intended public repository name, release tag, package hashes, release assets, and `gh` commands for creating the repository and release from the open-source zip. It does not create a GitHub repository, push code, publish a release, store GitHub credentials, or bypass the no-secret package review.

After the maintainer has created the public repository and release, import public publication evidence:

```bash
./scripts/import_github_publication_evidence.py --repo <owner>/short-drama-whitelabel-mobile --strict
```

The importer writes `build/github-publish/github-publication-evidence.json`. It records repository visibility, default branch, release tag, release assets, local open-source package hashes, and no-secret scan results. It uses GitHub CLI to read public metadata only; it does not store GitHub tokens or tenant credentials.

## CI Boundary

`.github/workflows/mobile-flutter.yml` runs:

- Flutter dependency resolution, analysis, tests, prototype layout overflow checks, native config validation, and mobile package secret scan.
- Manual workflow dispatch plus upload of open-source, store-assets, iOS CI, store-signing, store-publish-config, and store-submission-evidence handoff files.
- Android debug APK, release APK, and release AAB builds for `hongguo`, `douyin`, `hippo`, and `reelshort`, with uploaded release manifests and APK/AAB secret scans.
- Unsigned iOS debug and release builds for `hongguo`, `douyin`, `hippo`, and `reelshort` on GitHub-hosted macOS/Xcode runners.
- `ios-ci-evidence` import of the downloaded unsigned iOS artifacts into public Info.plist metadata, uploaded as `mobile-ios-ci-artifact-evidence`.

iOS store release still requires tenant-owned signing setup and Apple Developer capability enablement. The template script uses `--no-codesign` so CI or a tenant machine can validate unsigned iOS builds separately from App Store signing.
