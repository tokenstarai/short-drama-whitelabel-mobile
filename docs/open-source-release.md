# Open Source Release Notes

## Default License

The mobile template is prepared for Apache-2.0 release. Add the repository-level license before public GitHub publishing if the whole monorepo is released; otherwise keep `mobile/` as the licensed package boundary.

## What Can Be Published

- Flutter source under `mobile/lib`.
- Public sample configs under `mobile/assets/config`.
- Template docs under `mobile/docs`.
- Android/iOS platform template files, excluding ignored local/generated files.
- App-level iOS privacy manifest `ios/Runner/PrivacyInfo.xcprivacy`, while it stays limited to template-default public declarations.
- Shared iOS entitlements file `ios/Runner/Runner.entitlements` for Sign in with Apple, while tenant team ids, provisioning profiles, and certificates stay outside Git.
- Example prototype screenshots generated from synthetic posters and demo data.
- Original generated launcher icons for the five template flavors.
- CI checks for static analysis, widget tests, prototype layout overflow tests, native config validation, Android debug APK builds, Android release APK/AAB builds, release manifest generation, store handoff manifest generation, unsigned iOS debug/release flavor builds, and secret scanning.
- Store-assets package generated by `scripts/export_store_assets.py`, containing per-flavor listing drafts, localized copy, review notes, data-safety starter facts, and publish-safe screenshots for tenant store-submission handoff.
- iOS CI handoff package generated by `scripts/export_ios_ci_handoff.py`, containing public workflow trigger, unsigned artifact names, Xcode scheme/xcconfig paths, tenant Apple action checklist, and no-credential boundary.
- iOS CI artifact evidence generated by `scripts/import_ios_ci_artifacts.py` after downloading unsigned GitHub Actions `Runner.app` artifacts.
- Store-signing handoff package generated by `scripts/export_store_signing_handoff.py`, containing placeholder iOS export-options and Android signing templates without real signing material.
- Store-publish config package generated by `scripts/export_store_publish_config.py`, containing tenant-fillable App Store, Google Play, and Android direct release configuration templates without provider credentials.
- Store-submission starter package generated by `scripts/export_store_submission_starter.py`, containing public evidence input examples, operator checklists, per-flavor submission runbooks, a global operator runbook, and an offline evidence collector without signing material or signing/credential file references.
- Store-submission evidence template and guide generated by `scripts/import_store_submission_evidence.py`, containing only public TestFlight, Play internal, or direct-distribution status fields and no signing material or signing/credential file references.
- GitHub-ready template zip generated by `scripts/export_open_source_template.py`, after reviewing the package manifest and no-secret scan.

## What Must Not Be Published

- Tenant Edge HMAC secrets.
- OAuth client secrets.
- Stripe, PayPal, wallet, bank, or crypto private keys.
- Cloudflare API tokens, Stream signing keys, R2 credentials, or webhook signing secrets.
- Third-party app screenshots, copied marks, or trade dress assets.

Tenants must update `ios/Runner/PrivacyInfo.xcprivacy` before store submission if they add native analytics SDKs, advertising SDKs, tracking domains, new accessed API categories, or collection of data beyond the template defaults. They must also enable the Sign in with Apple capability on their own Apple Developer App ID to match `ios/Runner/Runner.entitlements`.

## Public Template Boundary

Templates are style presets, not brand clones:

- `hongguo_inspired`: light theater, red/orange charts, grid discovery.
- `douyin_inspired`: dark vertical feed, immersive player, compact unlock.
- `hippo_inspired`: channel theater, VIP emphasis, structured discovery.
- `reelshort_inspired`: overseas poster-forward drama, coins/subscription framing.

Tenants may override app name, colors, logo, supported locales, legal links, target region, and store compliance mode from the tenant portal export manifest.

The MVP product boundary is defined in [PRD Index](prd/README.md), including
the current-state audit, clickability matrix, tenant config parity, API-only
integration contract, iOS/Android parity, open-source package boundary, and
high-fidelity style-system brief.

The release manifest is written to `build/release-manifests/mobile-artifacts.json` and contains only public release metadata plus SHA256 checksums. The store handoff manifest is written to `build/release-handoff/mobile-store-handoff.json` and combines public flavor config, build commands, artifact references, legal URLs, compliance-filtered payment visibility, OAuth/deep-link callback registration metadata, App Store/Google Play product registration metadata, native capability registration metadata, store review declarations, distribution channel readiness, store submission metadata, publish-safe screenshot or WYSIWYG review asset references, and tenant-owned Apple/Google/direct-distribution checklist items. `paymentVisibility` shows configured providers, app-visible providers, providers hidden by the selected store compliance mode, and whether external payments are allowed. `authCallbackRegistration` lists each flavor's app URL scheme, `auth` host, enabled social OAuth callback URIs, expected callback query parameters, and native config files that own the scheme; provider credentials still stay in Tenant Edge/API Worker secrets. `storeProductRegistration` lists IAP or Play Billing product rows for store-compliant builds, the Tenant Edge receipt verification endpoint, and tenant actions for replacing template product ids while keeping App Store shared secrets and Google Play service-account credentials server-side. `nativeCapabilityRegistration` derives iOS and Android capability checklists from auth and payment providers, including Sign in with Apple, IAP, Play Billing, custom URL schemes, OAuth SHA fingerprints, direct-distribution signing, privacy manifest, entitlements, app icons, and native config paths. `storeReviewDeclarations` gives tenants a starter answer map for App Review, Play Console Data safety, and Android direct distribution declarations: digital-content payment policy, external payment visibility, account deletion, Sign in with Apple, user-generated content, privacy manifest, and direct-distribution payment duties. `distributionChannelReadiness` maps each flavor to its primary release channel, TestFlight build command and local Xcode blocker, Play internal AAB artifact, Android direct APK/AAB artifacts, and tenant-owned signing actions. `storeSubmission` gives tenants a public default listing starter, per-locale listing drafts for the configured supported locales, review notes, data-safety reminders, store-account actions, and either eight prototype screenshot paths or the CoolShow WYSIWYG review board with SHA256, size, and PNG dimensions without replacing final App Store Connect, Play Console, or direct-distribution questionnaires. `android_runtime_smoke.py` can add local emulator/device proof that all five release APKs install and launch; it writes only ignored runtime evidence under `build/runtime-smoke/`. `ios_build_matrix.py` adds unsigned iOS debug/release build evidence for all five flavors on a full-Xcode machine, or writes an ignored `blocked` report with exact missing-toolchain details. `ios_runtime_smoke.py` can add local Simulator proof that all five iOS simulator builds install and launch; by default it requires an already booted Simulator, and `--boot-simulator` must be used explicitly when an operator wants the script to boot one. `export_ios_ci_handoff.py` adds a public GitHub Actions handoff for remote unsigned iOS builds when local Xcode is unavailable; `import_ios_ci_artifacts.py` imports downloaded unsigned `Runner.app` artifacts into auditable public metadata. These CI and simulator artifacts still do not replace tenant TestFlight, App Store, or signing evidence. `export_store_signing_handoff.py` adds no-secret placeholder signing templates for tenant Apple/Google/direct distribution setup. `export_store_publish_config.py` adds no-secret tenant-fillable release configuration templates for App Store Connect, Play Console, and Android direct handoff. `export_external_account_handoff.py` adds a no-secret external account checklist for Apple Developer, Google Play, Android direct, OAuth/social login, C-end payments, and legal review, mirroring the tenant portal external setup entry and accepting only public status, links, ids, build numbers, callback URLs, and checksums. `export_store_submission_starter.py` adds no-secret operator runbooks that connect signing templates, publish config templates, the evidence collector, and the strict importer without creating passing evidence. `import_store_submission_evidence.py` adds a blocked-by-default public evidence template and guide for TestFlight, Play internal, or direct-distribution status after tenant signing is done; tenants must not paste certificates, provisioning profiles, upload keystores, service-account JSON, OAuth credentials, payment keys, webhook secrets, bank credentials, crypto keys, Cloudflare tokens, or file references such as `.p12`, `.p8`, `.mobileprovision`, `.jks`, `.keystore`, `AuthKey`, `upload-keystore`, or service-account files into it. `mobile_completion_closure.py` reruns the public import/audit chain after new CI or tenant store evidence lands and writes the current `canClaimComplete` decision plus remaining blockers; it can resolve the GitHub repository from `--repo`, `GH_REPO`, `GITHUB_REPOSITORY`, or git `origin`. Build outputs stay under ignored `build/` directories. Tenants must replace the template debug signing configuration with their own store signing setup outside Git before submission.

`capture_wysiwyg_previews.mjs` builds `lib/preview_main.dart`, captures release-rendered Flutter Web screenshots with Playwright/Chromium, and writes public WYSIWYG PNGs plus `build/wysiwyg-preview/wysiwyg-preview-manifest.json`. It captures all five templates across the eight MVP screens, waits for the Flutter runtime view before capture, and rejects likely blank PNGs through dimension and file-size checks. Set `CHROME_PATH=/path/to/Chrome` if the bundled Playwright browser is unavailable. These screenshots use generated demo assets and must not include third-party app screenshots, copied marks, tenant secrets, signing material, or provider credentials.

`export_ui_preview_gallery.py` writes `build/ui-preview-gallery/mobile-ui-preview-gallery.html`, SVG and PNG readable overview/contact sheet files, WYSIWYG runtime boards, `mobile-ui-preview-gallery.zip`, and `ui-preview-gallery-manifest.json`. It packages the five style presets and eight MVP screens from Flutter golden screenshots plus 40 release-rendered runtime captures, the five-template home board, the CoolShow eight-screen board, the full five-template/eight-screen board, 20-screen PNG review boards, and 40-screen PNG contact sheets so tenants can approve the visual template before replacing final branding and store screenshots.

`export_store_submission_starter.py` writes `build/store-submission-starter/mobile-store-submission-starter.zip`, per-flavor public evidence input examples, operator checklists, and `store-submission-evidence-collector.html`. The collector is an offline public metadata worksheet for preparing per-flavor files under `build/store-submission-evidence/flavors/`; the combined `store-submission-evidence.input.json` path is a fallback only when no per-flavor inputs are present. It must not contain signing files, provider credentials, service-account files, OAuth secrets, payment secrets, webhook secrets, bank credentials, wallet keys, or Cloudflare tokens.

`store_submission_evidence_preflight.py` writes a public preflight JSON/Markdown report under `build/store-submission-evidence/`. It checks per-flavor evidence files before the combined fallback input, is intended for tenant operators to review missing flavor evidence before running the strict importer, does not create passing evidence, and must not contain signing files or credentials.

`mobile_completion_audit.py` writes `build/completion-audits/mobile-app-completion.json` so release reviewers can distinguish completed template evidence from local iOS environment blockers. It also writes `build/release-handoff/mobile-tenant-release-package.json`, `build/release-handoff/mobile-tenant-release-package.md`, and `build/release-handoff/mobile-tenant-release-package.zip`, a public tenant handoff manifest, human-readable guide, and downloadable archive that links the store handoff manifest, Android artifact manifest, store assets package, UI preview gallery, WYSIWYG runtime boards, SVG/PNG review boards, iOS CI handoff, store signing handoff, store publish config templates, external account checklist, store submission evidence/template, top-level store-submission status summary, completion unblocker, store evidence fix queue CSV/Markdown, open-source docs, per-flavor release channels, OAuth callback metadata, store product metadata, tenant-required publishing actions, and no-secret boundaries.

## Store Submission Evidence

Run this after tenant-owned signing and store-track setup:

```bash
./scripts/import_store_submission_evidence.py
```

The first run writes `build/store-submission-evidence/store-submission-evidence.template.json`, `store-submission-evidence.guide.md`, and a blocked `store-submission-evidence.json`. Save per-flavor files under `build/store-submission-evidence/flavors/<flavor>.input.json`; use the guide for allowed statuses and required public checklist flags, fill only public TestFlight, Play internal, or direct-distribution status fields and public evidence references, then rerun the importer with `./scripts/import_store_submission_evidence.py --source-dir build/store-submission-evidence/flavors --strict`. Per-flavor input files take precedence over the combined input for preflight and source-dir strict import; `store-submission-evidence.input.json` remains available only as a fallback when no per-flavor inputs are present. Evidence refs can stay as plain public strings or use structured objects with `label`, `type`, and at least one of `value`, `url`, or `sha256`; allowed types are listed in the generated `publicEvidenceRefSchema.allowedTypes`. Structured `url` values must be public HTTPS URLs, so localhost, file URLs, plain HTTP, loopback, private IP ranges, and `.local` hosts are rejected. `evidenceCapturedAt` and structured ref `capturedAt` values must be timezone-aware ISO-8601 timestamps that are not in the future. This evidence remains ignored build output and must not contain signing files, provider credentials, service-account JSON, webhook secrets, bank credentials, crypto keys, or Cloudflare tokens.

## Completion Closure Refresh

Run this after importing new iOS CI artifacts or tenant store-submission evidence:

```bash
./scripts/mobile_completion_closure.py
```

Use `--run-id <run-id>` to pin a specific GitHub Actions run, `--store-submission-source <path>` for a tenant-filled public evidence file, and `--strict` when CI should fail unless `canClaimComplete` is true. If `--repo` is omitted, the script uses `GH_REPO`, `GITHUB_REPOSITORY`, or git `origin`; without a repository it refreshes local artifact evidence only. It always regenerates the store-submission starter package so `build/store-submission-starter/store-submission-operator-runbook.md` matches current signing and publish-config handoffs before importing evidence. The script writes `build/completion-closure/mobile-completion-closure.json` and `.md`; it contains command status, audit summary, and blocker next actions only. After writing the closure report, it refreshes `build/app-handoff/mobile-app-handoff.zip` and `build/release-handoff/mobile-tenant-release-package.zip` so tenant-facing archives include the same completion decision without secrets.

## GitHub Template Export

Run this from `mobile/` before publishing the mobile template as a standalone GitHub repository or release artifact:

```bash
./scripts/export_open_source_template.py
```

The script writes `build/open-source/short-drama-whitelabel-mobile.zip` and `build/open-source/open-source-template-manifest.json`. ```

The importer writes `build/github-publish/github-publication-evidence.json`. It records repository visibility, default branch, release tag, release assets, local open-source package hashes, and no-secret scan results. It uses GitHub CLI to read public metadata only; it does not store GitHub tokens or tenant credentials.

## CI Boundary

`.github/workflows/mobile-flutter.yml` runs:

- Flutter dependency resolution, analysis, tests, prototype layout overflow checks, native config validation, and mobile package secret scan.
- Manual workflow dispatch plus upload of open-source, store-assets, iOS CI, store-signing, store-publish-config, and store-submission-evidence handoff files.
- Android debug APK, release APK, and release AAB builds for `hongguo`, `douyin`, `hippo`, and `reelshort`, with uploaded release manifests and APK/AAB secret scans.
- Unsigned iOS debug and release builds for `hongguo`, `douyin`, `hippo`, and `reelshort` on GitHub-hosted macOS/Xcode runners.
- `ios-ci-evidence` import of the downloaded unsigned iOS artifacts into public Info.plist metadata, uploaded as `mobile-ios-ci-artifact-evidence`.

iOS store release still requires tenant-owned signing setup and Apple Developer capability enablement. The template script uses `--no-codesign` so CI or a tenant machine can validate unsigned iOS builds separately from App Store signing.
