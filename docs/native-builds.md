# Native Build Notes

## Generated Platform Projects

The template includes Flutter-generated Android and iOS platform projects:

- `android/` contains Gradle/Kotlin Android project files and four product flavors.
- `ios/` contains the Flutter Runner project and whitelabel xcconfig presets.

Generated local files are intentionally ignored and must not be committed:

- `android/local.properties`
- `ios/Flutter/Generated.xcconfig`
- `ios/Flutter/flutter_export_environment.sh`
- `ios/Flutter/ephemeral/`
- `ios/Runner/GeneratedPluginRegistrant.*`

These files are regenerated by Flutter on each developer machine.

`ios/Runner/PrivacyInfo.xcprivacy` is part of the publishable template. It is an app-level privacy manifest for the template shell and is referenced by the Runner target resources. Tenants must update it if they add native SDKs, tracking domains, accessed API categories, or data collection beyond the template defaults.

`ios/Runner/Runner.entitlements` is also part of the publishable template. It enables the Sign in with Apple entitlement for the shared Runner target, but it does not include tenant team ids, provisioning profiles, certificates, or signing material. Each tenant must enable the matching App ID capability in Apple Developer before TestFlight or App Store submission.

`build_flavor.sh` accepts:

- flavor: `coolshow`, `hongguo`, `douyin`, `hippo`, `reelshort`
- platform: `android`, `ios`
- mode: `debug`, `profile`, `release`
- Android package type: `apk` or `appbundle` as the optional fourth argument. The default is `apk`; `appbundle` is only valid for Android profile/release builds.

`write_release_manifest.sh` scans generated APK, AAB, and IPA artifacts under `build/` and writes `build/release-manifests/mobile-artifacts.json` with public release metadata: flavor, style template, platform, build mode, package type, app id, deep-link scheme, file size, and SHA256. The manifest is intentionally limited to public metadata and must not include tenant secrets or provider credentials.

`write_store_handoff_manifest.py` writes `build/release-handoff/mobile-store-handoff.json`. It combines public sample tenant config with the optional release artifact manifest, then lists each flavor's app metadata, legal links, auth/payment providers, compliance-filtered payment visibility, OAuth/deep-link callback registration metadata, App Store/Google Play product registration metadata, native capability registration metadata, store review declarations, distribution channel readiness, store submission metadata with per-locale listing drafts, publish-safe screenshot asset references with SHA256/size/dimensions, build commands, artifact paths, and tenant-owned Apple/Google/direct-distribution checklist items. The handoff file is public metadata only and must not include provider credentials, signing material, tenant HMAC credentials, or server-side Worker configuration.

`scan_release_artifacts.py` opens generated release APK/AAB packages and scans their contents for forbidden secret markers such as tenant HMAC secrets, Cloudflare tokens, OAuth client secrets, Stripe/PayPal secrets, webhook secrets, private keys, and raw playback manifest markers.

`android_runtime_smoke.py` installs and launches Android release APKs on a connected Android device or local emulator. `./scripts/android_runtime_smoke.py all --launch-emulator` uses the configured `HongGuo_AVD` when no device is attached, then writes `build/runtime-smoke/android-runtime-smoke.json` plus per-flavor launch screenshots under `build/runtime-smoke/screenshots/`. The runtime smoke evidence is public package/device/screenshot metadata only; it is optional for CI, but `mobile_completion_audit.py` validates it when present.

`build_coolshow_ios_test.sh` is the focused local path for a CoolShow iPhone preview build. It builds `com.coolshow.short`, packages `build/ios-test/coolshow/CoolShowShort-release-iphoneos-unsigned.ipa`, writes a public manifest, and, when a trusted iPhone plus Apple signing identity and `IOS_DEVELOPMENT_TEAM` are present, builds the `coolshow` scheme with automatic signing, installs it with `devicectl`, and launches the app on the phone. See `docs/coolshow-ios-test-install.md`.

`ios_build_matrix.py` builds unsigned iOS debug and release apps for all five flavors when full Xcode is selected. It writes `build/ios-build-matrix/ios-build-matrix.json` plus per-flavor Info.plist snapshots under `build/ios-build-matrix/app-info/`. If full Xcode is missing, it writes a `blocked` matrix with exact environment blockers. The evidence is public build metadata only; it must not contain Apple signing material, provisioning profiles, tenant secrets, OAuth secrets, payment secrets, or Cloudflare tokens.

`ios_runtime_smoke.py` builds iOS simulator apps, installs them on a booted Simulator, launches each flavor, and writes `build/runtime-smoke/ios-runtime-smoke.json` plus per-flavor launch screenshots under `build/runtime-smoke/ios-screenshots/`. By default it requires an already booted Simulator; pass `--boot-simulator` only when the operator explicitly wants the script to boot an available Simulator. The evidence is public simulator install/launch/screenshot metadata only and does not replace tenant TestFlight, App Store, or signing evidence.

`export_ios_ci_handoff.py` writes `build/ios-ci-handoff/mobile-ios-ci-handoff.zip` plus `build/ios-ci-handoff/ios-ci-handoff-manifest.json`. It reads the store handoff manifest and `.github/workflows/mobile-flutter.yml`, then emits per-flavor iOS CI metadata: workflow trigger, macOS/Xcode runner, unsigned artifact name, Xcode scheme, xcconfig, bundle id, deep-link scheme, expected Info.plist checks, the local `download_ios_ci_artifacts.py` import command, tenant Apple actions, and no-credential boundary.

`download_ios_ci_artifacts.py --repo <owner/repo>` uses GitHub CLI to download `mobile-<flavor>-ios-unsigned` artifacts for all five flavors into `build/ci-ios/`, then runs `import_ios_ci_artifacts.py --strict`. If `--repo` is omitted, it resolves `GH_REPO`, `GITHUB_REPOSITORY`, or git `origin`. By default it selects the latest successful `mobile-flutter.yml` run; pass `--run-id <run-id>` to import a specific run, and use `--dry-run` to print the exact `gh run download` and import commands without contacting GitHub. The script does not read or write Apple signing material, provisioning profiles, tenant secrets, OAuth secrets, payment secrets, or Cloudflare tokens.

`import_ios_ci_artifacts.py` reads downloaded GitHub Actions unsigned iOS artifacts from `build/ci-ios/` and writes `build/ios-ci-evidence/ios-ci-artifacts.json`. It records per-flavor `Runner.app` Info.plist metadata, bundle ids, display names, artifact names, app sizes, and snapshot hashes without Apple signing material. When no artifacts are present, it writes a blocked evidence file so the audit can distinguish missing CI artifacts from local Xcode blockers.

`export_store_assets.py` writes `build/store-assets/mobile-store-assets.zip` plus `build/store-assets/store-assets-manifest.json`. It extracts the public store-submission section from the store handoff manifest, copies all 40 publish-safe prototype screenshots, and emits per-flavor listing JSON with localized copy, review notes, data-safety starter facts, tenant-required actions, screenshot hashes, and no private provider configuration.

`capture_wysiwyg_previews.mjs` builds `lib/preview_main.dart` for each selected flavor, serves the local Flutter Web release build, captures 390px Playwright/Chromium screenshots, and writes them under `build/wysiwyg-preview/` with `wysiwyg-preview-manifest.json`. It captures all five templates across the eight MVP screens. The captures are public review evidence only and must not contain tenant secrets, provider credentials, signing material, or copied third-party app assets.

`export_ui_preview_gallery.py` writes `build/ui-preview-gallery/mobile-ui-preview-gallery.html`, SVG and PNG readable overview/contact sheet files, WYSIWYG runtime preview boards, `build/ui-preview-gallery/mobile-ui-preview-gallery.zip`, and `build/ui-preview-gallery/ui-preview-gallery-manifest.json`. It packages the same five-template/eight-screen Flutter golden screenshots into an offline HTML review gallery plus 40 release-rendered runtime captures, a five-template home board, a CoolShow eight-screen board, a full five-template/eight-screen board, and 20-screen/40-screen PNG review boards so tenants can approve the selected template before replacing branding and final store screenshots.

`export_store_signing_handoff.py` writes `build/store-signing-handoff/mobile-store-signing-handoff.zip` plus `build/store-signing-handoff/store-signing-handoff-manifest.json`. It emits per-flavor iOS `ExportOptions.plist.template` files, Android signing placeholder files, release artifact references, tenant Apple/Google/direct-distribution actions, and a no-signing-material boundary. It is a handoff checklist only; tenants must fill team ids, provisioning profiles, certificates, upload keystores, passwords, and store account configuration outside Git.

`export_store_publish_config.py` writes `build/store-publish-config/mobile-store-publish-config.zip` plus `build/store-publish-config/store-publish-config-manifest.json`. It packages per-flavor `publish-config.template.json` files for tenant-fillable App Store, Google Play, and Android direct release fields: app identity, legal URLs, OAuth callback URIs, store product ids, data-safety starters, enabled release channel, and server-side credential boundaries. It does not include signing files, provider credentials, webhook credentials, or service-account files.

`export_external_account_handoff.py` writes `build/external-account-handoff/mobile-external-account-handoff.json`, `build/external-account-handoff/mobile-external-account-handoff.md`, and `build/external-account-handoff/mobile-external-account-handoff.zip`. It mirrors the tenant portal "外部账号与签名资料接入入口", lists the public fields accepted for each release channel, and keeps signing files, provider credentials, webhook signing values, bank credentials, wallet keys, payment API keys, OAuth confidential values, Cloudflare tokens, and credential file names outside Flutter, GitHub, and tenant portal pages.

`export_store_submission_starter.py` writes `build/store-submission-starter/mobile-store-submission-starter.zip`, per-flavor public evidence input examples, operator checklists, per-flavor submission runbooks, `all-flavors/store-submission-evidence.input.example.json`, `store-submission-operator-runbook.md`, and `store-submission-evidence-collector.html`. The runbooks connect signing handoff templates, publish config templates, the offline collector, per-flavor evidence files under `build/store-submission-evidence/flavors/`, and the strict evidence import command. They are public metadata only and do not contain signing files, provider credentials, service-account files, OAuth secrets, payment secrets, webhook secrets, bank credentials, wallet keys, Cloudflare tokens, or signing/credential file references.

`store_submission_evidence_preflight.py` writes `build/store-submission-evidence/store-submission-evidence-preflight.json` and `build/store-submission-evidence/store-submission-evidence-preflight.md`. It checks per-flavor files under `build/store-submission-evidence/flavors/` first, falls back to the combined input file only when no per-flavor files are present, lists each flavor's missing public checklist fields, and never creates passing store-submission evidence by itself.

`import_store_submission_evidence.py` writes `build/store-submission-evidence/store-submission-evidence.template.json`, `build/store-submission-evidence/store-submission-evidence.guide.md`, and `build/store-submission-evidence/store-submission-evidence.json`. With no tenant input file, it writes a blocked evidence report. After tenant-owned signing and TestFlight, Play internal testing, or direct distribution are configured, save tenant-filled per-flavor files as `build/store-submission-evidence/flavors/<flavor>.input.json` and run `scripts/import_store_submission_evidence.py --source-dir build/store-submission-evidence/flavors --strict`; per-flavor input files take precedence over the combined input for preflight and source-dir strict import. A single `build/store-submission-evidence/store-submission-evidence.input.json` file is still supported only when no per-flavor inputs are present. The importer merges per-flavor files into the combined source before strict validation. Evidence refs can be plain public strings or structured objects with `label`, `type`, and at least one of `value`, `url`, or `sha256`; allowed object types are listed in `publicEvidenceRefSchema.allowedTypes`. Structured `url` values must be public HTTPS URLs, so localhost, file URLs, plain HTTP, loopback, private IP ranges, and `.local` hosts are rejected. `evidenceCapturedAt` and structured ref `capturedAt` values must be timezone-aware ISO-8601 timestamps that are not in the future. Do not include certificates, provisioning profiles, upload keystores, service-account JSON, OAuth credentials, payment keys, webhook secrets, bank credentials, crypto keys, Cloudflare tokens, or file references such as `.p12`, `.p8`, `.mobileprovision`, `.jks`, `.keystore`, `AuthKey`, `upload-keystore`, or service-account files.

`export_open_source_template.py` writes `build/open-source/short-drama-whitelabel-mobile.zip` plus `build/open-source/open-source-template-manifest.json` for GitHub publication review. It packages only publish-safe source, docs, synthetic assets, native scaffolding, and CI metadata while excluding build outputs, generated Flutter files, CocoaPods caches, local Android properties, signing material, env files, and secret-like values.

`export_github_publish_handoff.py` writes `build/github-publish/github-publish-manifest.json`, `github-publish-guide.md`, and `github-release-notes.md` with public repository metadata, package hashes, release assets, and `gh` commands for creating a standalone template repository and release. It does not create repositories, push code, publish releases, or store GitHub credentials.

`import_github_publication_evidence.py --repo <owner>/short-drama-whitelabel-mobile --strict` writes `build/github-publish/github-publication-evidence.json` after the public template repository and release exist. It records repo visibility, default branch, release tag, release assets, local open-source package hashes, and no-secret scan results from public metadata only.

`mobile_completion_audit.py` writes `build/completion-audits/mobile-app-completion.json`. It verifies source/native/CI structure, five template flavors, 40 prototype PNGs, expected test files, runtime C-end identity persistence, source secret boundaries, open-source package evidence, store assets package evidence, UI preview gallery evidence with WYSIWYG runtime screenshots plus SVG/PNG review boards, iOS static release configuration including the app privacy manifest and Sign in with Apple entitlements, iOS CI handoff package evidence, imported iOS CI artifact evidence, store signing handoff package evidence, store publish config package evidence, external account handoff package evidence, store-submission starter runbooks, store submission evidence, required iOS build matrix status, Android release manifest coverage, store handoff manifest coverage, tenant release package coverage, optional Android runtime smoke evidence when present, and local iOS build environment status. Missing full Xcode or CocoaPods is reported as an environment blocker by default; pass `--strict-ios` when a release gate must fail on that blocker.

The audit also writes `build/release-handoff/mobile-tenant-release-package.json`, `build/release-handoff/mobile-tenant-release-package.md`, and `build/release-handoff/mobile-tenant-release-package.zip`. This public package links the store handoff manifest, Android artifact manifest, store assets package, UI preview gallery, WYSIWYG runtime boards, SVG/PNG contact sheets, iOS CI handoff, store signing handoff, store publish config templates, external account handoff checklist, store-submission starter runbooks, store submission evidence/template, the top-level store-submission status summary, completion unblocker, store evidence fix queue CSV/Markdown, open-source docs, per-flavor release channels, OAuth callback metadata, store product metadata, and tenant-required actions. The zip is a downloadable handoff archive containing the public guide, manifests, completion unblocker, fix queue, WYSIWYG/PNG review boards, external account checklist, and existing handoff packages. It is designed for tenant publishing handoff only and must not contain signing material, provider credentials, Tenant Edge HMAC secrets, payment secrets, or Cloudflare tokens.

The repo-level wrapper `npm run infra:mobile-app-completion-audit` runs the mobile audit, verifies release APK/AAB secret scan output, and writes `artifacts/mobile-completion/mobile-app-completion-audit.json`, `artifacts/mobile-completion/mobile-app-completion-audit.md`, and `implementation/mobile-app-completion-audit.md`.

`check_app_config.mjs` validates that public tenant sample config, Flutter flavor constants, native bundle metadata, supported locales, and generated release manifests remain aligned. `check_native_config.sh` also verifies that iOS AppIcon selection stays in the flavor xcconfig files instead of being hardcoded in the Xcode project.

`resolve_flutter_bin.sh` is shared by the build/check scripts. It resolves Flutter in this order: explicit `FLUTTER_BIN`, `flutter` on `PATH`, `/tmp/flutter/bin/flutter`, then `$HOME/.local/flutter/bin/flutter`. This keeps local verification from accidentally using a stale SDK when a fresh SDK was bootstrapped elsewhere.

The player uses Flutter's `video_player` plugin for Tenant Edge authorized playback URLs. Social login handoff uses `url_launcher` to open tenant-hosted OAuth start URLs in the system browser, then `app_links` listens for the tenant app URL-scheme callback and completes the callback through Tenant Edge so provider token exchange stays server-side. Android already declares `INTERNET` and an `auth` deep-link host; iOS declares `$(APP_DEEP_LINK_SCHEME)` in `CFBundleURLTypes`. Full iOS playback and OAuth callback verification still require full Xcode plus CocoaPods because these Flutter plugins include native platform code.

## iOS Requirements

iOS builds require full Xcode, not only Command Line Tools, plus CocoaPods for Flutter plugin integration. `build_flavor.sh` preflights both before calling Flutter so local failures explain the missing toolchain instead of falling through to ambiguous project errors. The script passes `--no-codesign`; each tenant must configure their Apple team, signing certificates, provisioning profiles, App ID Sign in with Apple capability, and OAuth callback URLs in their own Apple developer account before App Store submission.

The template includes shared Xcode schemes for the template apps:

- `coolshow`
- `hongguo`
- `douyin`
- `hippo`
- `reelshort`

Each scheme points at the Runner target and selects the matching flavor xcconfig before build. The helper script runs an unsigned iOS build after copying the selected xcconfig into `ios/Flutter/WhitelabelDefaults.xcconfig`, so Flutter, Xcode, bundle id, URL scheme, and Dart runtime flavor stay aligned without storing Apple signing material in Git.

Each flavor also has its own generated AppIcon asset catalog:

- `AppIcon-hongguo`
- `AppIcon-douyin`
- `AppIcon-hippo`
- `AppIcon-reelshort`
- `AppIcon-coolshow`

Run `scripts/generate_launcher_icons.py` to refresh the original placeholder icons. Tenants can replace these files with their own brand assets, but copied third-party app icons, screenshots, or marks should not be committed to the open-source template.

`scripts/check_native_config.sh` can run without Xcode. It validates Android flavors, iOS xcconfig presets, bundle-id placeholders, deep-link placeholders, Sign in with Apple entitlements, flavor launcher icon assets, and native secret boundaries. Full iOS build verification still requires `xcodebuild`; after installing and selecting full Xcode, run `scripts/ios_build_matrix.py all` to generate the required local iOS build evidence.

If local full Xcode is unavailable, run `scripts/export_ios_ci_handoff.py` and trigger the GitHub Actions workflow on a repository with Actions enabled:

```bash
gh workflow run mobile-flutter.yml
gh run list --workflow mobile-flutter.yml --limit 5
gh run download <run-id> -n mobile-hongguo-ios-unsigned -D build/ci-ios/hongguo
./scripts/import_ios_ci_artifacts.py
```

Repeat the artifact download for `mobile-douyin-ios-unsigned`, `mobile-hippo-ios-unsigned`, and `mobile-reelshort-ios-unsigned`, then rerun `scripts/import_ios_ci_artifacts.py`. This proves the remote unsigned build path, but it does not replace the local completion gate: `scripts/ios_build_matrix.py all` must still pass on a full-Xcode machine before claiming full app completion.

After an unsigned iOS build passes, run `scripts/export_store_signing_handoff.py` and copy the relevant `ExportOptions.plist.template` to a tenant-controlled signing environment. Replace `TENANT_APPLE_TEAM_ID` and the provisioning profile placeholder outside Git, then run a tenant-owned signed archive/export workflow. The generated template keeps the app-store-connect export method and manual signing shape explicit without storing Apple credentials in the repo.

Run `scripts/export_store_publish_config.py` before store submission handoff. The generated per-flavor publish config templates tell tenants which public fields to fill in App Store Connect, Play Console, or direct-distribution operations after the signing setup is ready.

Run `scripts/export_store_submission_starter.py` before collecting tenant-owned store evidence. The generated starter package includes per-flavor input examples, checklists, submission runbooks, a global operator runbook, and an offline evidence collector HTML page that can be used to prepare per-flavor files under `build/store-submission-evidence/flavors/` without storing secrets. Start from `build/store-submission-starter/store-submission-operator-runbook.md` when handing a flavor to a tenant operator; the combined `store-submission-evidence.input.json` path is a fallback only when no per-flavor inputs are present.

Run `scripts/store_submission_evidence_preflight.py` while a tenant is filling store-submission evidence. The preflight report is safe to share because it lists public input paths, statuses, and blockers only; it is not a substitute for the strict importer.

Run `scripts/import_store_submission_evidence.py` after the tenant has produced signed builds and configured TestFlight, Play internal testing, or direct distribution. The first run creates a blocked template plus `store-submission-evidence.guide.md`; full completion remains blocked until all per-flavor files under `build/store-submission-evidence/flavors/` record public tenant-owned store status for all five tenant release flavors without secrets. Per-flavor input files take precedence over the combined input for preflight and source-dir strict import; `store-submission-evidence.input.json` is a fallback only when no per-flavor inputs are present. The guide lists allowed status values, required public checklist flags, evidence examples, and the strict re-import commands.

Run `scripts/export_completion_unblocker.py` when the completion audit is still blocked. It writes `build/completion-unblocker/mobile-completion-unblocker.json` and `.md` with the external action plan for full Xcode, unsigned iOS CI artifact import, store-submission starter/runbook refresh, tenant store-submission evidence import, field-level remediation rows for each blocked store-submission input, and the final `npm run infra:mobile-app-completion-audit` rerun. It also writes `mobile-store-evidence-fix-queue.csv` and `.md`, a compact queue that flattens every blocked field into `flavor`, `inputPath`, `field`, `tenantAction`, `acceptedPublicEvidence`, and `forbiddenEvidence` for tenant operator assignment. The unblocker is public metadata only and does not contain signing files or provider credentials.

Run `scripts/mobile_completion_closure.py` after new GitHub Actions iOS artifacts or tenant store evidence are available. It automatically downloads unsigned iOS artifacts with `gh` when `--repo`, `GH_REPO`, `GITHUB_REPOSITORY`, or git `origin` identifies the repository; pass `--run-id <run-id>` to pin a workflow run. If no repository is available, it refreshes local artifact evidence only. It always regenerates the store-submission starter package first so `build/store-submission-starter/store-submission-operator-runbook.md` matches the current handoff files. Pass `--store-submission-source <path>` to import tenant-filled public store evidence. It then reruns the completion audit and writes `build/completion-closure/mobile-completion-closure.json` plus `.md` with `canClaimComplete` and any remaining blockers, and refreshes the app handoff plus tenant release package so the tenant-facing zip contains the same closure decision. Use `--strict` only when the calling workflow should fail if completion is still blocked.

Run `scripts/export_github_publish_handoff.py` after `scripts/export_open_source_template.py` when the template is ready for a public GitHub repository or release. It writes `build/github-publish/github-publish-manifest.json`, `github-publish-guide.md`, and `github-release-notes.md` with public `gh` commands, package hashes, and release assets, but it does not create repositories, push code, or store GitHub credentials.

## Android Requirements

Android builds require JDK 21, a configured Android SDK with the platform, build tools, accepted SDK licenses, and NDK version requested by the active Flutter Gradle plugin. Do not commit `android/local.properties`; each developer machine or CI runner should provide `JAVA_HOME`, `sdk.dir`, and `flutter.sdk` locally.

On a new machine, run `flutter doctor -v` first, then install the versions it reports. For Flutter 3.44, the current SDK requirement is Android platform 36 plus NDK `28.2.13676358`:

```bash
flutter doctor -v
brew install openjdk@21
export JAVA_HOME=/opt/homebrew/opt/openjdk@21/libexec/openjdk.jdk/Contents/Home
sdkmanager --licenses
sdkmanager "platforms;android-36" "build-tools;36.0.0" "ndk;28.2.13676358"
```

If the NDK download is interrupted, delete the partial NDK directory before retrying. A malformed NDK install commonly fails Gradle with a missing `source.properties` file.

Avoid building with JDK 26 or newer until the Android Gradle toolchain explicitly supports it. On this machine, JDK 26 failed during `JdkImageTransform`; switching to Homebrew `openjdk@21` produced the template debug APKs.

`scripts/build_flavor.sh` now prefers local JDK 21/17 paths for Android builds even when the shell has a newer `JAVA_HOME`. Set `SHORT_DRAMA_KEEP_JAVA_HOME=1` only when intentionally testing a custom JDK.

For store handoff, tenants should build Android App Bundles and then replace the template debug signing configuration with their own Play/App Signing setup outside the repository:

```bash
./scripts/build_flavor.sh hongguo android release appbundle
./scripts/write_release_manifest.sh
./scripts/write_store_handoff_manifest.py
```

The template release build currently uses the debug signing config only so CI and local machines can verify the package pipeline without storing tenant signing material in Git.
`scripts/export_store_signing_handoff.py` also emits `android-signing.properties.template` files per flavor. Treat them as field names only; resolved keystore paths, aliases, and passwords belong in tenant CI secrets or local keychains, not repository files.

Android flavor-specific launcher icons live under:

```text
android/app/src/<flavor>/res/mipmap-*/ic_launcher.png
```

Gradle resolves these resources ahead of the shared `main` icon for matching flavor builds.

## CI Coverage

`.github/workflows/mobile-flutter.yml` supports manual dispatch and runs the mobile secret scan, Flutter analysis, Flutter tests, native config validation, handoff/signing/publish-config package upload, Android debug APK, release APK, release AAB builds plus release manifests for all five template flavors, unsigned iOS debug plus unsigned iOS release builds for all five template flavors, and an `ios-ci-evidence` job that downloads `mobile-*-ios-unsigned` artifacts, imports their public Info.plist metadata, and uploads `mobile-ios-ci-artifact-evidence`. Store signing, Apple Developer capability enablement, TestFlight upload, and App Store submission remain tenant-owned because Apple teams, provisioning profiles, OAuth callback ownership, and merchant/payment capabilities differ per tenant.

## Secret Boundary

Native project files may contain public bundle ids, app names, URL schemes, and Dart defines. They must never contain tenant HMAC secrets, OAuth client secrets, payment provider secrets, Cloudflare tokens, Stream signing keys, webhook signing secrets, crypto private keys, or bank credentials.
